# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Graph-RAG: a RAG pipeline that will eventually build a graph aware of contradictions between sources. The project is split into four phases; **phase 1** (what currently exists in this repo) is a vanilla RAG pipeline using LangChain/ChromaDB, built around a fixed research topic:

> "Do larger language models actually reason better, or just pattern-match better?"

Phase 1 plan (see [Readme.md](Readme.md)):
1. Fetch arXiv papers matching the topic, download each PDF, extract its text directly (no external conversion service).
2. Chunk each PDF's text (page-aware, fixed-size with overlap).
3. Embed chunks and push them into ChromaDB along with metadata (article id, page numbers) and the raw chunk text.
4. Build an evaluation dataset and test retrieval.
5. Wire an LLM/LangGraph on top of the retrieval layer, including a citation-graph expansion step.

The Readme's own running notes describe this as a project that got "way more overly complicated" and is being deliberately simplified back down â€” GROBID and the local/fallback dual-embedding-server setup were both removed for that reason (see below).

**Phase 1 is now complete**, including the two Gradio UIs (see the Gradio UIs section below) and deployment: the LangGraph agent server runs on LangSmith (LangGraph Platform), and both Gradio apps are deployed as Hugging Face Spaces â€” see [Readme.md](Readme.md)'s Status section. Local dev (`uv run langgraph dev` + running each UI module directly) still works exactly as documented throughout this file; the deployed versions just point `settings.langgraph_client_url` at the hosted LangSmith deployment instead of `http://localhost:2024`.

## Pipeline architecture

`start_topic_ingestion(topic, collection_name, max_chars, chunk_overlap, num_of_pdfs, concurrency, force)` in [src/make.py](src/make.py) is the real entrypoint (used by `__main__`) â€” it consults the topics registry (see the Topics registry section below) before delegating to `run_pipeline(topic, collection_name, max_chars, chunk_overlap, num_of_pdfs, concurrency)`, the lower-level fetchâ†’process function. Call `run_pipeline` directly only if you deliberately want to bypass the registry checks. Chunking size is now caller-configurable per topic (`max_chars`/`chunk_overlap` flow all the way through to `DocumentPipeline`'s `chunk_pdf_blocks` call, see below) rather than fixed defaults â€” `__main__` range-validates them (`num_of_pdfs` in `[10, 50]`, `max_chars` in `[1000, 10000]`, `chunk_overlap <= max_chars // 2`) before calling `start_topic_ingestion`, raising a bare `Exception` on violation. There are **no on-disk checkpoints** â€” everything from PDF download through chunk storage happens in memory, per paper, inside one async pipeline. A failed run must be restarted from scratch (fetch included); the only safety net is per-document retry-once-then-skip inside `DocumentPipeline`, so one bad paper doesn't abort the whole run.

1. **Fetch** ([src/utils/fetch.py](src/utils/fetch.py) â€” `FetchPDFs.fetch()`): queries the arXiv API (`arxiv` package), filters to results with a reachable `pdf_url`, and returns a `list[MetaData]` directly (no CSV). Raises if fewer than 10 reachable results are found. Before fetching, `src/make.py`'s `generate_search_query(topic)` sends the fixed topic to an LLM (`init_chat_model`, `CHAT_MODEL_ID`, a few-shot prompt) to produce a concise, keyword-focused arXiv search query â€” `FetchPDFs` is given that generated query, not the raw topic verbatim.

2. **Per-paper pipeline** ([src/pipeline.py](src/pipeline.py) â€” `DocumentPipeline.__process_one`): for each paper, concurrently (bounded by `asyncio.Semaphore`, default 8) and fully async:
   - download the PDF into memory (`httpx.AsyncClient`, never written to disk)
   - chunk it directly from the raw PDF bytes via **PyMuPDF** ([src/utils/chunking.py](src/utils/chunking.py) â€” `chunk_pdf_blocks(pdf_bytes, max_chars, overlap)`): opens the bytes with `pymupdf.open(stream=..., filetype="pdf")`, walks each page's text blocks (`page.get_text("blocks")`), skips image blocks, and buffers text across blocks/pages into chunks up to `max_chars` with a suffix/prefix `overlap` carried into the next chunk â€” no separate GROBIDâ†’Markdown conversion step, no header-aware splitting anymore. Each chunk is a `Chunk` dict (`schema.py`): `{"text": str, "pages": set[int]}` â€” the set of page numbers the chunk's text spans, since a chunk can straddle a page boundary.
   - chunks and the paper's `title + "\n\n" + abstract` are embedded separately, each as part of its own store's `.add()` call (`VectorStore`'s attached `HuggingFaceRouterEmbeddingFunction` embeds `documents=` automatically) â€” no manual embedding call in the pipeline itself.
   - **build a per-article citation graph**: via the injected `semantic_scholar_client` ([src/clients/semantic_scholar.py](src/clients/semantic_scholar.py) â€” `SemanticScholarClient`) and [src/utils/citation_graph.py](src/utils/citation_graph.py)'s `build_citation_graph`, look up the paper on Semantic Scholar (by its cleaned-up arXiv id, `clean_arxiv_id`) and build a one-hop `networkx.DiGraph` of its references + citing papers (edges point citing â†’ cited, node id = Semantic Scholar `paperId`), capped at `max_references`/`max_citations` (default 10 each) â€” Semantic Scholar returns every reference/citation uncapped, and some papers have hundreds, which was blowing past chroma's per-metadata-field size limit on the serialized graph. Returns the graph plus the paper's own `semantic_scholar_id` (the center node's `paperId`), stored separately (see `PaperMetadata` below). A 404 (not indexed), an unparseable/old-style arXiv id, or hitting Semantic Scholar's rate limit yields an empty graph (and empty id) without failing the paper; any other error propagates like any other step's failure. This happens *before* either store write below (a hard failure here must still be safe to retry the whole per-paper chain without risking a duplicate-id write on the chunk store). A dedicated `s2_semaphore` (capacity 1) plus a `3s` pacing sleep throttles these calls, independent of the wider per-paper concurrency â€” Semantic Scholar's **unauthenticated public endpoint has been observed to `429` immediately** in testing, so set `SEMANTIC_SCHOLAR_API_KEY` in `.env` for real runs (sent as an `x-api-key` header when present).
   - store the chunks in a `{collection_name}_chunks` Chroma Cloud collection, then the paper's metadata (including the serialized citation graph and its own `semantic_scholar_id`) in a `{collection_name}_papers` collection (both via `VectorStore.add`, wrapped in `asyncio.to_thread` since chromadb's client has no async API). Chunk ids are `<arxiv_id>-<i>` (index into that paper's chunk list) â€” no filename sanitizing needed since nothing ever becomes a file; chroma accepts arbitrary strings as ids.
   - **storage is per-paper atomic**: metadata and chunks for a paper are only written once every prior step for that paper succeeded â€” a paper that fails partway (bad PDF parse, no chunks extracted, embedding error, hard Semantic Scholar failure) gets neither a `papers` nor a `chunks` entry, so the two collections can't go inconsistent with each other. Other papers in the same run are unaffected.
   - on any exception, the whole per-paper chain (download included) is retried once, then logged and skipped (doesn't abort other papers' processing)

## Topics registry ([src/utils/topics.py](src/utils/topics.py))

Both `topic` and `collection_name` are supplied by the caller (there's no auto-derivation/slugification from topic text) â€” the risk that creates is a `collection_name` typo/reuse silently merging two unrelated topics' data via `get_or_create_collection`, plus the ordinary cost of re-ingesting a topic that's already there under slightly different wording. `start_topic_ingestion` (see above) guards both, backed by a dedicated `topics` Chroma collection (one row per topic: id = `collection_name`, document = the topic text itself, embedded like any other document):

- `find_collection_name(collection_name, registry)` â€” exact-id lookup (`registry.get().get(ids=[collection_name], ...)`). If it returns a topic that doesn't match the one being ingested, `start_topic_ingestion` refuses outright â€” this check is **not** bypassable with `force`, since forcing it would just mean silently merging unrelated data into one collection pair.
- `find_similar_topic(topic, registry, threshold)` â€” semantic dedup check via `registry.get().query(query_texts=[topic], n_results=1, include=["documents", "distances"])`, guarding the empty-registry case (`.count() == 0`) before querying. Returns a `TopicMatch` (`collection_name`/`topic`/`distance`) if the top hit is within `threshold`, else `None`. If a match is found, `start_topic_ingestion` logs it (pointing at the existing `collection_name` to reuse) and skips ingestion â€” **unless** `force=True`, which proceeds anyway.
- The distance returned is **L2, not cosine similarity** â€” `HuggingFaceRouterEmbeddingFunction` doesn't override `EmbeddingFunction.default_space()`, so Chroma's default `"l2"` space applies. `src/make.py`'s `TOPIC_SIMILARITY_THRESHOLD` (currently `0.4`) was calibrated once empirically (see `tests/topics.py`): a reworded near-duplicate topic landed at `~0.27`, unrelated topics at `~1.3`-`1.4`. Revisit this constant if real usage disagrees â€” it's not a principled value, just an observed gap.
- `register_topic(topic, collection_name, metadata, registry)` writes the row. `metadata` is a `TopicMetadata`: `created_at` (ISO timestamp) plus the actual `num_of_pdfs`/`max_chars`/`chunk_overlap` values that ingestion ran with â€” recorded so a later "which settings did this topic get ingested with" question doesn't require reading logs. `start_topic_ingestion` builds this dict itself (from its own params, not by reading anything back off `run_pipeline`) and calls `register_topic` **only after `run_pipeline` succeeds** â€” mirroring `DocumentPipeline`'s own per-paper atomic-write principle, so a failed ingestion (e.g. `FetchPDFs.fetch()`'s <10-reachable-PDFs `RuntimeError`) never leaves a registry entry pointing at empty `_papers`/`_chunks` collections.
- `list_topics(registry) -> list[dict]` enumerates every registered topic (`registry.get().get(include=["documents", "metadatas"])` with no `ids`/`where` filter, zipping `ids`/`documents`/`metadatas` into `{"collection_name", "topic", **TopicMetadata}` dicts). Chroma's native `Collection.get()` with no filter returns every row â€” this is the only enumeration primitive, `VectorStore` itself has no `list()`. Both Gradio UIs share this one helper (see Gradio UIs section below): the chat UI's topic dropdown and the pipeline UI's collections table.
- **Now wired up**: `src/agent.py` no longer hardcodes `"llm_chunks"`/`"llm_papers"` â€” see the RAG agent section below for the `context_schema`-based per-topic routing that replaced it. The chat UI passes whichever `collection_name` the user selected from `list_topics(...)` per chat session.

## Storage layout

Nothing is written to local disk â€” `src/clients/store.py`'s `VectorStore` connects to a **Chroma Cloud** database (`CloudClient`, via `CHROMA_API_KEY`/`CHROMA_TENANT_ID`/`CHROMA_DB_NAME` in `.env`). Collections, per `collection_name`:

```
{collection_name}_papers       # one row per fetched paper: title+abstract, title/pdf_url/citation_graph/semantic_scholar_id metadata
{collection_name}_chunks       # one row per chunk: article_id/pdf_url/page_no metadata, chunk text
{collection_name}_test         # eval set from tests/eval_vectordb.py: query, article_id/pdf_url/chunk metadata
topics                          # one row per ingested topic (not per-collection_name-prefixed): id=collection_name, document=topic text, see Topics registry below
```

**Collection names are no longer settings-driven** â€” there's no `Settings.collection_name` field anymore:
- [src/make.py](src/make.py)'s `__main__` hardcodes `collection_name = "llm"` and passes it into `start_topic_ingestion` (see Topics registry below), which â€” after its registry checks pass â€” calls `run_pipeline`, which derives `f"{collection_name}_papers"`/`f"{collection_name}_chunks"`. Both Gradio UIs' ingestion form take `collection_name` as a real user input instead (see Gradio UIs below), so `"llm"` is just the `__main__` script's own default, not a repo-wide constant.
- [src/agent.py](src/agent.py) no longer hardcodes any collection name â€” it derives `f"{collection_name}_chunks"`/`f"{collection_name}_papers"` per-invocation from `runtime.context.collection_name` (a `context_schema`-routed value, see the RAG agent section below), the same `_papers`/`_chunks` suffix convention as `run_pipeline`, just resolved dynamically instead of fixed at import time.

- [src/schema.py](src/schema.py) defines: `MetaData` (`id`, `title`, `pdf_url`, `abstract` â€” an arXiv fetch result, used across `src/utils/fetch.py`/`src/pipeline.py`), `Chunk` (`text`, `pages: set` â€” a chunker's raw output, pre-storage), `ChunkMetadata` (`article_id`, `pdf_url`, `page_no: list` â€” a chunk row's metadata; note this replaced the old `chunk_index: int` field, so a chunk's position is now recorded as which page(s) it came from, not its sequence number within the paper), `PaperMetadata` (`title`, `pdf_url`, `citation_graph`, `semantic_scholar_id` â€” a paper row's metadata; `semantic_scholar_id` is new, it's the paper's own citation-graph center-node id, used by the agent's `expand_related_papers` node), `EvalMetadata` (`article_id`, `pdf_url`, `chunk` â€” an eval-set row's metadata), `TopicMetadata` (`created_at`, `num_of_pdfs`, `max_chars`, `chunk_overlap` â€” a topics-registry row's metadata, recording what a topic was actually ingested with), `TopicMatch` (`collection_name`, `topic`, `distance` â€” not chroma metadata, just `find_similar_topic`'s return shape), `IngestionOutcome` (`status: Literal["ingested", "conflict", "skipped_similar", "failed"]`, `collection_name`, `topic`, `conflicting_topic: str | None`, `match: TopicMatch | None`, `error: str | None` â€” `start_topic_ingestion`'s return type, see below; also not chroma metadata, never stored, just a function-contract shape), and an `ExtractedContent` alias. The `*Metadata` TypedDicts type chroma metadata dicts at construction time (`src/pipeline.py`, `src/utils/topics.py`, `tests/eval_vectordb.py` once fixed); reading them back uses `typing.cast(...)` since chromadb's own read-side type is a generic untyped mapping (`src/agent.py`, `tests/eval_vectordb.py`).
- `start_topic_ingestion` (see Pipeline architecture above) returns an `IngestionOutcome` instead of `None` â€” every one of its four exit paths (registered-under-a-different-topic conflict, similar-topic-found skip, `run_pipeline` raising, or a clean success) now returns a distinguishable typed result instead of a caller-invisible early `return`. `run_pipeline` failures are caught *inside* `start_topic_ingestion` (a `try/except Exception`, not `except:` â€” so `asyncio.CancelledError` still propagates) rather than left for the caller to catch, so the Gradio pipeline UI never needs its own `try/except` around the call. This doesn't change the existing atomic-write guarantee (`register_topic` is still only reached after `run_pipeline` succeeds) â€” it just adds a reporting path around the same control flow. `src/make.py`'s own `__main__` ignores the return value, so this is backward compatible there.
- Chunk metadata carries both `article_id` (the true arXiv id, e.g. `http://arxiv.org/abs/2307.08072v2` â€” the same value as `MetaData["id"]`) and `pdf_url` (e.g. `https://arxiv.org/pdf/2307.08072v2`) as separate fields â€” `article_id` is the cross-reference between a ChromaDB chunk and its source paper row. A paper's `citation_graph` metadata field is a JSON string (`src/utils/citation_graph.py`'s `serialize_graph`/`deserialize_graph`, via `networkx.node_link_data`) â€” chroma metadata values must be scalars, so the graph is never stored as a native object.
- `src/clients/store.py`'s `VectorStore.add()` types `metadatas` loosely as `Sequence[Mapping[str, object]]` (not chromadb's own narrower `Metadata` type) so that any of the domain `*Metadata` TypedDicts above can be passed without a `cast` at every call site â€” pyright can't verify a TypedDict against chromadb's `Metadata` (`Mapping[str, <narrow scalar union>]`) directly (it widens a TypedDict's value type to `object` for that check regardless of its actual field types), so there's a single `# type: ignore[arg-type]` on the internal `self.collection.add(...)` call instead of scattering casts across `src/pipeline.py`, `tests/vector_store.py`, and `tests/eval_vectordb.py`.

## RAG agent ([src/agent.py](src/agent.py))

A **4-node** LangGraph graph (was 3-node before the citation-graph expansion step was added), exported as `graph` and wired up in [langgraph.json](langgraph.json) as the `agent` graph (path updated to `./src/agent.py:graph`). State is a pydantic `BaseModel`: `query`, `search_query`, `chunks: list[Chunk]`, `related_papers: list[RelatedPaper]`, `answer`. `Chunk` (agent-local pydantic model, distinct from `schema.py`'s `Chunk` TypedDict of the same name) has `text`/`article_id`/`pdf_url`/`page_no: list`. `RelatedPaper` has `article_id`/`pdf_url`/`content`.

1. `rewrite_query` â€” sends the raw `query` to an LLM (`CHAT_MODEL_ID`, few-shot prompt) to produce a keyword-rich `search_query` better suited to vector retrieval than a conversational question.
2. `retrieve_chunks` â€” queries the `chunks` ChromaDB collection with `query_texts=[search_query]` for the top 5 matches, reading each result's metadata via `typing.cast(ChunkMetadata, ...)` into a `Chunk`.
3. `expand_related_papers` â€” **new**: takes the retrieved chunks' source papers, pulls each one's stored one-hop citation graph (`PaperMetadata["citation_graph"]`, deserialized), and unions every neighbor node id (minus each paper's own `semantic_scholar_id`, so a paper never counts as its own neighbor) into a candidate set. Looks those candidates up in the `papers` collection via a `where={"semantic_scholar_id": {"$in": [...]}}` metadata filter (citation-graph nodes are Semantic Scholar `paperId`s, not the arXiv-based row ids used elsewhere, so this can't just be an `ids=` lookup), drops any candidate that's already one of the directly-retrieved chunks' own source papers, and stores the rest as `RelatedPaper`s (title+abstract text only, no chunk-level excerpt).
4. `generate_answer` â€” sends the original `query` (not `search_query`) plus two separately-labeled context sections to the LLM: `Context` (the directly retrieved chunk excerpts â€” primary evidence, cited by `pdf_url`) and `Related papers` (the citation-graph neighbors from step 3 â€” explicitly instructed to be used only for broader background/framing, not as evidence for a specific granular claim, since it's title+abstract only with no excerpt-level detail; still cited by `pdf_url`).

`src/agent.py` reads its config from `settings` and builds its `model` once at module scope, reused across graph invocations. **Per-topic collection routing** (added alongside the Gradio UIs â€” see below) uses LangGraph's `context_schema`/`Runtime` mechanism rather than the older, now-deprecated `config`/`config["configurable"]` path (deprecated since LangGraph v0.6.0, removed in v2.0.0 â€” confirmed against the installed package source, not just docs):
- `AgentContext` is a `@dataclass` with one field, `collection_name: str` (the base name, e.g. `"llm"` â€” nodes append `_chunks`/`_papers` themselves, same convention as `run_pipeline`). `builder = StateGraph(State, context_schema=AgentContext)`.
- `retrieve_chunks(state: State, runtime: Runtime[AgentContext])` and `expand_related_papers(state: State, runtime: Runtime[AgentContext])` both take a `runtime` param and read `runtime.context.collection_name` (attribute access â€” `AgentContext` is a dataclass, matching LangGraph's own documented `Runtime` example). `rewrite_query`/`generate_answer` are unchanged (they never touch a store) â€” LangGraph allows nodes with different signatures in the same graph.
- `_get_chunks_store(collection_name)`/`_get_papers_store(collection_name)` are `@lru_cache`d module-level functions that build (and cache, keyed by `collection_name`) the `VectorStore(...).get()` `chromadb.Collection` handle â€” without the cache, every chat message would open a brand-new Chroma Cloud connection.
- The chat UI (`src/UI/agent_ui.py`) calls the graph via `langgraph_sdk`'s stateless runs (`thread_id=None` â€” `State` has no cross-turn memory field, so no LangGraph thread/checkpointer is needed; conversation history lives only in the Gradio UI's `gr.BrowserState`), passing `context={"collection_name": ...}` per invocation. Confirmed live (against a real running `langgraph dev` + Chroma Cloud) that this correctly routes each request to that topic's collections, node-update events (`stream_mode="updates"`) arrive in the fixed order `rewrite_query â†’ retrieve_chunks â†’ expand_related_papers â†’ generate_answer`, and `generate_answer`'s answer streams token-by-token via `stream_mode="messages-tuple"` (wire event names are `"metadata"`, `"updates"`, `"messages"` â€” not the literal `"messages-tuple"` string requested).

## Gradio UIs ([src/UI/](src/UI/))

Two independent Gradio apps, run/import as `src.UI.agent_ui`/`src.UI.pipeline_ui` (namespace package, same `-m`-from-root convention as the rest of `src/`). Neither is wired into `langgraph.json` â€” only the agent graph itself is; the UIs are plain Gradio processes that talk to it (or to `src.make` directly) from the outside. Deployed as two separate Hugging Face Spaces (see the top of this file / Readme.md's Status section); locally each runs via `uv run python -m src.UI.<name>` on its own port (7860/7861).

### `src/UI/agent_ui.py` â€” chat UI

- Connects via `langgraph_sdk.get_client(url=settings.langgraph_client_url, api_key=settings.langgraph_client_api or None-if-unset)` â€” `langgraph_client_api` is only passed through when non-empty, so local `langgraph dev` (no auth) still falls back to the SDK's own `LANGGRAPH_API_KEY`/`LANGSMITH_API_KEY`/`LANGCHAIN_API_KEY` env-var auto-detection instead of having it explicitly disabled.
- Topic selection: a `gr.Dropdown` populated from `list_topics(...)` (see Topics registry above). Selecting one and clicking "New chat" creates a session dict `{id, title, collection_name, topic, messages, created_at}` and appends it to browser-side state.
- **Session persistence is `gr.BrowserState`** (Gradio 5+'s component that transparently syncs a Python value to the browser's `localStorage`, restoring it automatically on page reload â€” scoped per-browser/per-origin, nothing server-side or DB-side). This is what satisfies "chats live only in the browser, restored on reload, never touch a DB, invisible to any other browser."
- Each session's `collection_name` is frozen in its own dict at creation time and every chat turn reads it from there (`respond()`'s `graph_client.runs.stream(..., context={"collection_name": session["collection_name"]}, ...)`) â€” **not** from the topic dropdown's live value. The topic dropdown is therefore never disabled/locked; it only ever controls what a *new* "New chat" click will use next, decoupled from whichever session is currently displayed. (An earlier version disabled the dropdown whenever any session was active to "prevent changing topic mid-conversation" â€” that was redundant, since the freeze already happens structurally, and it had the side effect of permanently blocking topic selection for any subsequent new session once one existed.)
- Streaming: `stream_mode=["updates", "messages-tuple"]`. `"updates"` events surface which node just ran (mapped through `NODE_LABELS` into a "Processing: â€¦" label). `"messages"` events are filtered to `metadata.get("langgraph_node") == "generate_answer"` before appending to the answer (since `rewrite_query` also calls the LLM and would otherwise leak its tokens into the visible answer).

### `src/UI/pipeline_ui.py` â€” ingestion UI

- Form fields exactly match `start_topic_ingestion`'s user-facing params: `topic`, `collection_name`, `num_of_pdfs` ([10,50]), `max_chars` ([1000,10000]), `chunk_overlap` (`<= max_chars // 2`, recomputed live via the max-chars field's `.change()` handler). `_validate_ingestion_params` in this file replicates `make.py`'s `__main__`-only bounds validation (duplicated, not factored into `make.py`, to keep that file's diff minimal).
- Submit handler directly `await`s `start_topic_ingestion` (no background task queue/polling) and reads its `IngestionOutcome` (see Storage layout/schema below) to render one of four status messages. A `skipped_similar` outcome reveals an "Ingest anyway" button that re-invokes the same handler with `force=True`.
- `gr.Dataframe` table lists every registered topic via `list_topics(...)` â€” refreshed on load and after every submit resolves. Unlike the chat UI's per-browser sessions, this table (and the underlying `topics`/`_papers`/`_chunks` collections) is genuinely shared/global â€” any user/browser sees the same data, since it's sourced from the real Chroma Cloud `topics` collection, not browser state.
